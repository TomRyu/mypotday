왜 이 문서가 중요한가

계(ROSCA)는 수령 순번이 빠를수록 미래 납부 의무가 크고, 도덕적 해이 유인이 존재합니다. 플랫폼이 이 비대칭성을 제도로 해소하지 못하면 계 파탄 → 사용자 신뢰 붕괴 → 서비스 종료로 이어집니다. 예방·탐지·사후 대응을 레이어드(layered) 방어 체계로 구축해야 합니다.

SECTION 01

문제의 구조적 본질

계(ROSCA)는 본질적으로 시간 차 신용 시스템입니다. 모든 리스크 설계는 이 구조적 비대칭성을 전제로 해야 합니다.

STRUCTURAL ASYMMETRY
1회차 수령자:  실질적으로 (N-1)개월분 무이자 차입 → 도덕적 해이 유인 최대
마지막 수령자: 실질적으로 (N-1)개월분 무이자 대출 공여 → 피해 최대

예시: 10명 계, 월 100만원
  ├─ 1번 수령자: 900만원 미래 납부 의무 (도망 시 계원들 900만원 손실)
  └─ 10번 수령자: 이미 900만원 납부, 100만원 남음 (피해는 미미하나 계 파탄 경험)

결론: 순번 1~3번 수령자의 도주가 가장 치명적.
        플랫폼이 이 구간을 집중 관리해야 한다.
한국 속칭 "먹고 튀기" — 계를 수령하고 이후 불입을 중단하는 행위. 전통 계에서도 반복적으로 발생해온 패턴으로, 디지털 플랫폼에서는 더 익명성이 높아 리스크가 증가합니다.
SECTION 02

예방 단계 — 시작 전에 막는 설계

A. 신원·신용 인증 (가입 시)
조치내용중요도
실명 인증 본인 명의 계좌 연동 필수 — 타인 명의 계좌로 가입 불가 필수
신용점수 조회 나이스·올크레딧 연동, 최소 신용점수 기준선 설정. 점수별 가입 가능 계 규모 차등 필수
금융거래 이력 최근 2년 내 연체·불이행 기록 확인. 기록 있을 시 후순위 강제 배정 중요
플랫폼 내 이력 기존 계 납부 실적을 스코어화. 신규 계 가입 시 신용 등급 상향 반영 장기
핵심 설계 원칙: 신용점수가 낮을수록 → 수령 가능 순번 후순위 강제 배정, 또는 고액 계 가입 제한. 신뢰도가 검증된 계원이 먼저 받는 구조가 자기선택(self-selection) 효과를 만든다.
B. 자동이체 선(先) 설정 의무화 (계 시작 시)

계 가입 후 첫 불입일 전에 자동이체 등록을 완료해야만 참여가 확정되도록 설계합니다.

ONBOARDING FLOW
가입 신청
  ↓
자동이체 계좌 등록 완료   ← 이 단계 미완료 시 참여 불가
  ↓
곗주 승인
  ↓
계 시작 확정

자동이체 실패 시:
  D+1  → 앱 푸시 + 카카오 알림톡 즉시 발송
  D+3  → 재출금 시도 + 수동 납부 요청
  D+7  → 플랫폼 개입 + 강제 조치 예고
C. 보증금 에스크로 예치 — 핵심 설계

수령 전 보증금을 플랫폼 에스크로에 예치하는 방식이 가장 강력한 단일 예방책입니다. 도망쳐 봤자 보증금을 잃기 때문에 합리적 행위자라면 불이행을 선택하지 않습니다.

ESCROW EXAMPLE — 10명 계, 월 100만원
1회차 수령 예정자 (가장 리스크가 큰 포지션)
  잔여 납부 예정액 = 9개월 × 100만원 = 900만원
  에스크로 보증금 예치 = 잔여액의 30% → 270만원 예치 후 수령 가능
  정상 납부 완료 시 → 만기 후 전액 반환

보증금 구간별 차등 (제안 기준)
  순번 1번 = 잔여 납부액의 30%
  순번 2번 = 잔여 납부액의 20%
  순번 3번 = 잔여 납부액의 15%
  순번 5번~ = 잔여 납부액의 5~8%
  마지막 2번 = 면제 (이미 리스크 없음)

신용점수 우대: 신용 A등급 → 보증금 비율 50% 할인
부가 수익 연계: 에스크로로 모인 자금을 단기 MMF·파킹통장 등으로 운용 시 이자 수익 발생. 법적 검토(전자금융거래법, 자본시장법) 후 반영 가능.
D. 순번 배정 방식 설계
배정 방식특징리스크 수준
무작위 추첨 공평하나 신용도 미반영 높음
입찰 (낙찰계) 고이자 납부자 우선, 시장원리 적용 중간
신용 기반 우선 신뢰도 높은 사람 먼저 수령 → 자기선택 효과 낮음
플랫폼 보증 후순위 플랫폼이 선지급 보증, 미납자 추심 대행 최저
SECTION 03

탐지 단계 — 조기에 문제를 포착하는 시스템

D+0
자동이체 출금 시도일
정상 출금 완료 시 납부 확인 알림 발송. 출금 실패 시 즉시 다음 단계 진입.
D+1
출금 실패 감지
앱 푸시 알림 + 카카오 알림톡 즉시 발송. "오늘 자동이체가 실패했습니다. 00일까지 납부해주세요."
D+3
재출금 시도 + 곗주 알림
재출금 자동 시도 + 곗주에게 해당 계원 연체 알림. 계원에게 카카오톡/SMS 추가 발송.
D+7
플랫폼 고객센터 개입
플랫폼이 해당 계원에게 직접 연락. 납부 의향 확인. 법적 조치 예정 안내 시작.
D+10
법적 조치 사전 경고문 발송
내용증명 발송 검토. "14일 이내 미납 시 법적 절차를 개시합니다." 공식 경고.
D+14
에스크로 충당 + 법적 절차 개시
보증금 에스크로에서 미납액 자동 충당 → 다른 계원 보호. 법적 조치 공식 착수.
의도적 불이행 조기 식별 신호: 연락 두절 + 자동이체 계좌 잔액 0 + 카드 거래 중단이 동시 발생 → 의도적 도주 패턴으로 판단, 즉시 D+14 법적 단계로 전환.
SECTION 04

사후 대응 — 불이행 발생 시 대응 체계

A. 즉각 보전 (D+14 ~ D+30)

가장 먼저 다른 계원의 피해를 최소화합니다.

① 에스크로 보증금 선충당 → 다른 계원 피해 즉시 최소화
② 보험금 청구 → 계파산보험 가입 계의 경우 보험사에 청구
③ 책임 분담 기준 집행 → 약관에 명시된 곗주 vs 플랫폼 책임 비율 적용
B. 법적 조치 수단 비교
수단내용소요기간비용
지급명령 법원 독촉, 이의 없으면 판결과 동일한 효력. 가장 빠른 채권 회수 수단 2~4주 인지대 소액
민사소송 채권 강제집행, 판결 후 재산 압류·경매 가능 6개월~1년 변호사 비용
형사 고발 처음부터 납부 의사 없었을 경우 사기죄(형법 §347) 적용 가능. 플랫폼 로그가 핵심 증거 수개월 0원
신용정보 등록 연체 정보 신용정보원 제출 → 신용점수 하락. 즉각적이고 강력한 억지력 즉시 0원
형사 고발 성립 요건: 계 가입 당시부터 납부 의사가 없었음을 입증. 플랫폼의 채팅 로그, 열람 기록, 계좌 잔액 변동 데이터가 핵심 증거로 활용됩니다. 반드시 데이터를 서버에 보존해야 합니다.
C. 플랫폼 내 블랙리스트 시스템
BLACKLIST SYSTEM
불이행 확정 시:
  ① 플랫폼 계정 즉시 정지
  ② 실명 기반 블랙리스트 DB 등록
  ③ 동일인 재가입 차단 (본인인증 연동 — 주민번호/CI 값 기반)
  ④ 업계 공유 블랙리스트 장기 검토 (개인정보 보호법 제17조 검토 필요)

데이터 보존 정책:
  → 불이행 관련 로그: 최소 5년 보관 (법적 소멸시효 대응)
  → 개인정보: 회원 탈퇴 후에도 분쟁 해결 목적으로 보존 가능 (약관 명시 필수)
SECTION 05

비즈니스 모델 연계 — 리스크를 수익으로 전환

이 문제는 단순 피해 방지가 아니라 새로운 수익 라인으로 설계할 수 있습니다.

REVENUE LINE 1
계 파산 보험 중개
불입 불이행 시 잔여 계금 보장 보험. 월 보험료 계금의 0.3~0.5%. 플랫폼 커미션 10~15%. 기존 수익 모델의 "보험 중개" 라인과 직결.
REVENUE LINE 2
신용 등급별 차등 수수료
신용 A등급 계원 포함 계 → 수수료 0.8%. C등급 포함 계 → 수수료 1.3% + 보험 강제 가입. 리스크 프리미엄을 수수료에 반영.
REVENUE LINE 3
에스크로 운용 수익
보증금 에스크로 자금을 단기 MMF·파킹통장으로 운용. 연 2~4% 수익 발생 가능. 전자금융거래법·자본시장법 검토 선행 필요.
마케팅 포인트: "MyPotDay는 계 파산 보험이 적용됩니다" — 보험의 존재 자체가 신뢰 신호(trust signal)로 작동. 경쟁사 대비 최강의 안전성 차별화 포인트.
SECTION 06

약관 설계 핵심 항목

법적 집행력을 위해 가입 약관에 반드시 명시해야 할 항목입니다. 법률 검토 후 확정 필요.

  • 제1조 자동이체 실패 시 대체 납부 의무 및 기한 (실패 후 7일 이내 수동 납부)
  • 제2조 7일 이상 연체 시 플랫폼 에스크로 보증금 충당에 동의함 (계 가입 시 포괄 동의)
  • 제3조 14일 이상 연체 시 플랫폼의 법적 조치(지급명령, 민사소송, 형사 고발) 위임에 동의
  • 제4조 불이행 확정 시 신용정보기관 연체 정보 등록에 동의
  • 제5조 계 수령 후 불입 의무는 계 만기일까지 면제 불가하며, 수령 행위가 계속 납부 확약으로 간주됨
  • 제6조 분쟁 해결 목적으로 계 관련 데이터를 계 종료 후 5년간 보관함에 동의
SECTION 07

구현 우선순위 로드맵

즉시
자동이체 의무화 + 약관 정비 MVP 개발 전 설계에 반영. 법률 검토 병행. 추가 비용 없이 리스크 대폭 감소.
Phase 1 · 론칭 시
실명인증 + 신용조회 연동 + 에스크로 보증금 나이스·올크레딧 API 계약. 에스크로 계좌 설계. 핵심 예방 인프라 완성.
Phase 2 · MAU 1천명
계 파산 보험 파트너십 체결 하나손보·DB손보 등 핀테크 친화 보험사 접촉. 보험 상품 공동 설계. 수익 모델 확장.
Phase 3 · MAU 1만명
AI 기반 불이행 예측 모델 + 내부 신용 스코어링 납부 패턴·계좌 잔액·플랫폼 행동 데이터 기반 예측. 고위험 계원 자동 식별 및 사전 대응.
SECTION 08

창업자 질문 — 심층 토론

이 문서를 작성하며 실제로 나눈 질문과 답변입니다. 비즈니스 실현 가능성에 대한 솔직한 검토를 담았습니다.

Q
먹튀 문제를 해결하기 힘들기 때문에 이 비즈니스는 실패할 것 같군요.
A

전통 계는 이미 수백 년간 이 문제를 해결해왔습니다. 디지털 이전에도 계는 존재했고, 먹튀는 항상 있었습니다. 그런데도 계는 사라지지 않았습니다. 전통 계는 사회적 압력(동네 망신, 관계 파탄)으로 해결했습니다. 플랫폼은 그걸 시스템(에스크로, 자동이체, 법적 집행)으로 대체하는 것입니다. 해결 수단이 약해진 게 아니라 강해진 겁니다.

먹튀 리스크는 은행 대출 리스크보다 작습니다. 은행은 신원도 모르는 사람에게 무담보로 대출을 줍니다. 계 플랫폼은 실명 인증 + 계좌 연동 + 에스크로 + 데이터 증거까지 갖춥니다.

진짜 질문은 "먹튀를 막을 수 있는가"가 아니라 "막는 비용이 수익보다 크냐"입니다. 에스크로 보증금만 제대로 설계되면 먹튀 발생 시에도 플랫폼이 직접 손실을 지지 않습니다. 리스크가 0이 아니어도 경영 가능한 리스크(manageable risk) 수준으로 내려오면 사업은 됩니다.
Q
Esusu, Tanda, StepLadder(영국), 인도 Chit Fund — 이곳들의 자세한 운영 상황을 알고 싶어요.
A

실제 조사 결과는 일부 예상과 다릅니다. 솔직하게 정리합니다.

Esusu (미국) — 중요 정정
실제로는 ROSCA 앱이 아닙니다. 원래 계 개념으로 시작했으나 임차료 신용 보고 서비스로 완전히 피벗했습니다. 월세를 제때 내면 신용점수에 반영해주는 서비스. 사용자 1,200만 명, 기업가치 $1.2B(유니콘), 2025년 Series C $50M 투자 유치.

Q
쉽지 않아 보이네요...
A

솔직히 말씀드리면, 맞습니다. 쉽지 않습니다. 하지만 "쉽지 않다"와 "불가능하다"는 다릅니다. 그리고 쉬운 사업에는 경쟁자가 몰립니다.

현실적으로 리스크를 낮추는 방법이 하나 있습니다. 처음부터 돈을 다루지 않는 것입니다.

RISK-MINIMIZED LAUNCH STRATEGY
1단계: 계 관리 앱 (돈을 절대 안 건드림)
  → 순번 관리, 납부 확인 알림, 계원 채팅
  → 계주가 직접 돈을 걷고 보내는 기존 방식 유지
  → 플랫폼은 그냥 "스마트 계 수첩"
  → 먹튀 리스크 = 0 (플랫폼과 무관)
  → 핀테크 라이선스 = 불필요

2단계: 신뢰 이력 쌓인 후 에스크로 도입
  → 신뢰 이력이 있는 계원부터 시범 적용
  → 먹튀 데이터도 쌓이고, 대응 체계도 검증됨

3단계: 보험·금융 연계
  → 계 파산 보험, 신용 스코어링, 파생 서비스 확장
카카오뱅크도 처음에는 메신저였습니다. 토스도 처음에는 송금 UI였습니다. 첫 버전이 완벽한 핀테크일 필요가 없습니다. 1단계만으로도 한국에 존재하지 않는 서비스입니다.
